Maintainer's manual

Keeping the framework alive

How to review submissions, publish entries and manage images for the Constructability Framework — written for someone who has never opened these tools.

Every task is a real worked example, start to finish, with a picture of what should be on your screen at each step and exactly what to click next.

If you can follow instructions, you can run this.

The big picture

What you're actually maintaining

The framework is a website (index.html) that reads its entire contents from one file — data.json — plus a folder of images. That's it. You never edit the website's code. Your whole job is to keep data.json and the images/ folder correct and up to date.

Three small tools do that work. Each is a single web page you open on your own computer — they run entirely in your browser, upload nothing, and only read or write the files you hand them. Here's how everything fits together:

The routine

The loop you'll repeat

Whenever submissions have built up, you run the same six steps. The rest of this manual is just each of these explained slowly, with a full example. This is the whole job on one screen.

  1. Download the responsesIn Google Forms, export each form's responses as a CSV — up to three files (new entries, edits, comments).
  2. Work the inbox in the Data ManagerLoad data.json and the CSVs, then go through each submission: de-identify it, tidy it, and accept or reject it.
  3. Publish the draftsAccepted entries wait as hidden drafts. Review them together, then publish and download the updated data.json.
  4. Prepare the imagesRun any new photos through the Image Formatter to rename, shrink and strip location data, then drop them in images/.
  5. Run the DoctorPoint the Framework Doctor at your updated data.json and images/ folder to catch anything missing before it goes live.
  6. Commit & pushPush data.json, the new images and CHANGELOG.md to GitHub. The live site updates itself.

The main tool · walkthrough

Data Manager

Turn a raw form response into a properly structured, published entry — reviewed, de-identified and checked. We'll follow one real submission, “Off-form concrete upturns near obstructions”, from the inbox to accepted. Do it once with this example and you'll know the pattern for every entry after it. Each screen below is a faithful rebuild of the actual tool, not a photo, so it stays accurate as the tool is tweaked.

It has five tabs. You'll mostly use two: Submission inbox (new responses) and Drafts (accepted, waiting to publish). The others are Browse entries, Triage map and Rejected submissions.

Open the Data Manager

Get the tool running in your browser.

The tool lives in the repository at tools/framework-data-manager.html. Open it in your browser — double-clicking works, since it runs entirely offline. You'll land on “1 · Load data.json”, with “Load data.json to begin.” at the bottom.

👉 Open tools/framework-data-manager.html

Load the current data.json (and the form responses)

Give the tool the live content plus the new submissions to process.

First hand it the current data.json from the repository. Then drop in the response CSVs from Google Forms — up to three (new entries, edits, comments), any order. Each box only accepts its own file type, and the tool works out which form each CSV came from. A Loaded files list confirms what's in, and anything already processed is skipped.

Read it — and de-identify as you go

Make sure nothing identifying a person, company or site reaches the live page.

This is the one non-negotiable rule of the whole job. As you read each field, remove anything identifying: project, company, client or designer names, site addresses, and title blocks in images. The submitter's contact details sit in a separate box that is never published — but their free-text answers can still contain names, so read every field before you accept.

If in doubt, take it out

Rewriting “on the Bligh Street job for Acme Builders” as “on a recent project” loses nothing useful and protects everyone. Keep the engineering detail; strip the identity.

👉 Scan every field for names, addresses and title blocks

Check the dimension and confirm the id

File the entry under the right dimension and lock in its permanent address.

Each entry belongs to one of the five dimensions. Ours is a construction-method question, so it goes under Construction Methods. (If the submitter chose “Not sure”, you pick it here.) The tool generates a stable id from the name and warns if it's taken. Once an entry is live, never change its id — it would break saved links and related-entry connections.

Fill the design question, background and parameter

The words a reader sees at the top of the entry.

These arrive pre-filled from the form. Read them, tidy the wording, de-identify. The design question is what the entry answers; the background explains why it matters; the design parameter is the single measurable thing the thresholds are based on. You never write HTML — type, press Enter for a line break, and use B/I (or Ctrl/⌘+B) to bold or italicise.

framework-data-manager.html — entry editor
Design question
When designing concrete upturns near obstructions, what clearance is required for conventional formwork, and at what point should alternative systems be considered?
Design parameter
Available clearance between upturn face and nearest obstruction (mm)
Background
Conventional off-form concrete upturns require practical access to both sides of the formwork for placement, bracing, tying, bolt-up, stripping and inspection. This assumption breaks down when upturns are positioned close to existing fences, retaining walls, boundaries or other obstructions…

👉 Read, tidy and de-identify each field

Set the threshold bands

The heart of the entry: what each range of the parameter means for buildability.

Each band is a range of the design parameter, a rating, and its guidance. The submitter provides the guidance; you fill in the range for each band. The four ratings carry fixed colours — Best practice, Consider, Concern, Action required — so severity reads at a glance. A live preview shows exactly how it will look on the site.

framework-data-manager.html — threshold bands
≥ 1000 mmBest practice
Full access for formwork placement, bracing, bolt-up, and stripping. Both coil tie and threaded bar systems feasible.
700 – 1000 mmConsider
Coil tie system feasible but access is tight — a person cannot comfortably work behind the upturn. Consider whether alternatives would be simpler.
350 – 700 mmConcern
Coil ties cannot be used (plywood + timbers + bolt = 700 mm minimum). Threaded bar only, tightened by reaching in at arm's length. Only for upturns/walls up to ~1200 mm high.
< 350 mmAction required
Conventional double-sided off-form construction is unlikely to be feasible without redesign, specialist one-sided forming, temporary access from the adjacent side, or an alternative construction system.

👉 Fill the range for every band · check the colours read correctly

Add search keywords, element and stage

Help people find the entry, and tag what it applies to.

Search already works on everything written in the entry, so keywords are a top-up: words a searcher might type that aren't already in the text — trade slang (“reo”), other names (“upstand”, “nib”, “kerb”, “parapet”), or the plain way someone describes the problem (“no room to form”). Then tag the element(s) and lifecycle stage(s) it applies to.

Preview it

See the entry exactly as a visitor to the site will.

Press Preview on the card to open the live-site view. This is your last read-through: check the wording, that the bands make sense in order, and — one more time — that nothing identifying slipped through. If something's off, close the preview and fix the field.

Accept it

Send the reviewed entry to Drafts, ready to publish.

Happy with it? Press Accept. The card marks itself “✓ Will be applied” and the entry moves to the Drafts tab — accepted, but not yet on the live site. (Press Reject instead and you can add a reason that stays with the record, then send it back to the inbox later — nothing is ever forced through.)

Your entry is now a Draft

You've turned one form response into a clean, reviewed entry. It sits in Drafts, invisible to the public. When you've worked the whole inbox, you review the drafts together, publish them, and download the updated data.json — then prepare images and run the Doctor before committing.
