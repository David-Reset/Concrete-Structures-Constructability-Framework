Maintainer's manual

Keeping the framework alive

How to review submissions, publish entries and manage images for the Constructability Framework — written for someone who has never opened these tools.

Every task is a real worked example, start to finish, with a picture of what should be on your screen at each step and exactly what to click next.

Everything you need is in one place.

The big picture

What you're actually maintaining

The framework is a website (docs/index.html) that reads its entire contents from one file — docs/data.json — plus a folder of images. That's it. You never edit the website's code. Your whole job is to keep data.json and the docs/images/ folder correct and up to date.

Three small tools do that work. Each is a single web page you open on your own computer — they run entirely in your browser, upload nothing, and only read or write the files you hand them. Here's how everything fits together:

Before you start

Where the files live — and how to get them

Everything you maintain lives in one place: the GitHub repository that the live site is served from. That repository is the single source of truth. Your session always starts by taking a copy of a file out of it, and ends by putting an updated copy back.

The repo has two folders that matter to you, and they do different jobs:

  • docs/ — this is the published website. Everything the public sees is served straight out of this folder:
    • docs/data.json — all the content: every entry, threshold, comment and rubric. The file you load into the tools and commit back.
    • docs/images/ — every photograph and drawing the entries point at.
    • docs/index.html — the site itself. You never edit this.
    • docs/CHANGELOG.md — the running record of what changed and when.
    • docs/manual/ — this manual.
  • tools/ — the three maintainer tools and the three rubrics. Not inside docs/, because they aren't part of the public site — they're pages you open on your own machine.

Getting the current data.json

The Data Manager can't fetch anything by itself — it's an offline page, and you hand it the file. So before every session, get a fresh copy:

  1. Open the repository on GitHubThe framework repository — the same repo the live site is published from.
  2. Open the docs folderdata.json is not at the top level. It's inside docs/, along with the site itself. Click into docs first — then click data.json.
  3. Use the download buttonGitHub shows the file as a preview — don't copy the text out of the page. Use the download raw file button (the ⤓ icon, top-right of the file box) to save the real file. Do the same for docs/CHANGELOG.md if you want your changes appended to it.

The direct link, if you'd rather skip the clicking: docs/data.json on GitHub.

If you work with git, git pull in your local clone does the same job and is less fiddly — the file is simply there at docs/data.json on disk.

Always start from a fresh copy

This is the one mistake that quietly destroys other people's work. The Data Manager exports a whole, complete data.json — not a patch. If you start from a copy you downloaded last month and someone has published entries since, your export will not merge with theirs; it will replace them, silently reverting everything they added. Download data.json again at the start of every session, even if you're sure nothing has changed.

And where it goes back

When you're finished, the updated docs/data.json, any new files in docs/images/, and the updated docs/CHANGELOG.md are committed and pushed back to the same repository — back into docs/, where they came from. The live site rebuilds itself from them — there is nothing else to deploy. That's the last step of the loop below.

The routine

The loop you'll repeat

Whenever submissions have built up, you run the same six steps. Each tool's own pages explain these in depth, with a full worked example.

  1. Download the responsesIn Google Forms, export each form's responses as a CSV — up to three files (new entries, edits, comments).
  2. Work the inbox in the Data ManagerLoad data.json, then in the Submission inbox drop the response CSVs, decide on each submission, and click Apply decisions.
  3. Publish the draftsAccepted entries wait as hidden drafts. Review them together, then publish and download the updated data.json.
  4. Prepare the imagesRun any new photos through the Image Formatter to rename, shrink and strip location data, then drop them in docs/images/.
  5. Run the DoctorPoint the Framework Doctor at your updated data.json and docs/images/ folder to catch anything missing before it goes live.
  6. Commit & pushPush data.json, the new images and CHANGELOG.md to GitHub. The live site updates itself.

The main tool

Data Manager

The tool you'll live in. It turns raw form responses into properly structured, de-identified entries in data.json — the file the live site reads.

It's a single web page. You open it on your own computer; it runs in your browser, uploads nothing, and only touches the files you hand it. People submit ideas, edits and comments through the Google Forms; the Data Manager helps you review each one and turn the good ones into clean entries.

The five tabs

Everything happens across five tabs at the top of the tool. You'll spend nearly all your time in the first two:

  • Submission inbox — new form responses waiting to be reviewed. A badge shows how many.
  • Drafts — entries you've accepted but not yet published. They're invisible to the public until you publish.
  • Browse entries — everything already live, for editing or reordering.
  • Triage map — the trigger that decides which jobs each entry is suggested on, and a dry run to test it. Chapter 8.
  • Rejected submissions — things you turned down, kept in case you change your mind.

Work through it in order

These chapters follow one real submission — “Off-form concrete upturns near obstructions” — the whole way from the inbox to published. Do it once with this example and you'll know the pattern for every entry after. Start at chapter 1. Chapter 8 then covers the one thing that isn't about an entry's content at all: whether a designer is ever shown it.

Chapter 3

Reviewing a new entry

De-identify the submission, set its dimension and id, then fill the design question, background and parameter.

De-identify as you read

Make sure nothing identifying a person, company or site reaches the live page.

This is the one non-negotiable rule of the whole job. As you read each field, remove anything identifying: project, company, client or designer names, site addresses, and title blocks in images. The framework is anonymous by design. The submitter's contact details sit in a separate box that is never published — but their free-text answers can still contain names, so read every field before you accept.

If in doubt, take it out

Rewriting “on the Bligh Street job for Acme Builders” as “on a recent project” loses nothing useful and protects everyone. Keep the engineering detail; strip the identity.

Check the dimension and confirm the id

File the entry under the right dimension and lock in its permanent address.

Each entry belongs to one of the five dimensions. Ours is a construction-method question, so it goes under Construction Methods. (If the submitter chose “Not sure”, you pick it here.) The tool generates a stable id from the name and warns if it's already taken. Once an entry is live, never change its id — it would break saved links and related-entry connections.

Fill the design question, background and parameter

The words a reader sees at the top of the entry.

These arrive pre-filled from the form. Read them, tidy the wording, de-identify. You never write HTML — type, press Enter for a line break, and use B/I (or Ctrl/⌘+B) to bold or italicise.

Chapter 4

Threshold bands

The heart of an entry: the ranges of the design parameter, their ratings, and the colour scale a reader sees at a glance.

Threshold bands are the heart of an entry. Each band is a range of the design parameter, a rating, and the guidance shown for that range. The submitter provides the guidance; you fill in the range for each band. The four ratings carry fixed colours, so a reader sees the severity at a glance:

Set the range for each band

Turn the submitter's guidance into ordered, colour-coded ranges.

Work from best to worst. Each band's colour is fixed by its rating — Best practice, Consider, Concern, Action required — so you only choose the numbers. A live preview beside each field shows how it will look on the site.

Chapter 5

Reviewing with an AI

An optional second opinion. You hand entries to any AI along with the rubric that defines “good”, and it hands them back improved — or rated. You review everything before it changes a thing.

This is the one part of the tool that involves something outside your browser, so it's worth being clear about how it works. The Data Manager never talks to an AI itself. It writes a .json file; you give that file to whatever AI you like; you drop the reply back in. Nothing is uploaded, nothing is automatic, and nothing is saved until you export — exactly like the rest of the tool.

Two different reviews

There are two buttons and they do genuinely different jobs. Don't mix them up:

  • 🤖 Entry quality review — is the entry well framed? One clear question, one parameter you can act on, bands that follow from it, the right dimension, useful search keywords. It hands back the improved entry. Use it on new submissions while they're still in the inbox.
  • 🤖 Evidence & triage review — how well-supported is the guidance, and when should it surface on a job? It hands back a rating and a set of triage triggers, which you apply field by field. Use it on entries that are already in data.json.

The line an AI review will not cross

A quality review may reframe an entry — reword the question, tighten the parameter, rename a band, move it to a better dimension, add keywords. It must not re-author the guidance: it won't move a threshold from 200 mm to 250 mm, change what a band means, or invent evidence. Whether a number is right is a question for the evidence rubric and for people who know the answer — not for a reframing pass. If an AI ever hands back a changed threshold, reject it.

Teach the tool what “well framed” means.

You only ever do this once: the rubric is saved inside data.json and travels with the file, so it's baked into every request from then on. Re-drop the file to update it if the rubric changes.

A shortcut for a single entry

You don't have to open the review window at all. Expand any new-entry card in the inbox and click 🤖 Send to AI review — it downloads a request for just that one entry. The reply comes back into the same card. Handy when one submission is clearly a mess and the rest are fine.

The evidence & triage review

The second button, 🤖 Evidence & triage review, works on the same principle — load the rubrics once, pick entries, download a request, drop the reply back — but it answers different questions, and it lands somewhere different at the end. Use it on entries that are already in data.json, once the wording has settled.

Load the two rubrics — once

Teach the tool how to rate evidence, and when an entry should surface.

This review applies two rubrics, so there are two boxes: drop evidence-rating-rubric.md into one and triage-rating-rubric.md into the other. Like the quality rubric, both are saved inside data.json and travel with the file — so you do this once, not once per session, and a colleague who opens your data.json already has them. The tool warns you in orange if either is missing: without them the AI is guessing at a standard it has never been shown.

Apply it suggestion by suggestion

You accept each change on its merits — not the reply as a package.

Every suggestion sits beside the entry's current value with a tick-box. Read, edit the wording if you want, untick anything you disagree with, then Apply ticked changes. The evidence summary is the sentence a reader will see on the entry, so edit it here rather than accepting AI phrasing you wouldn't have written. If the AI proposed new facts for the questionnaire, they're reviewed in their own section first — with a warning if a proposed question hangs off a gate that doesn't exist — because a trigger can't be applied until the fact it names exists. Anything it couldn't match to a real entry id is listed separately and ignored. And as always: nothing reaches data.json until you export.

👉 Untick what you don't buy · Apply ticked changes · export

Re-drop a rubric whenever it changes

The rubrics live inside data.json — that's what makes a review repeatable and portable, and it's also the one way to get a quietly wrong answer. If you edit evidence-rating-rubric.md or triage-rating-rubric.md on disk, the copy baked into data.json does not change with it, and every request keeps shipping the old rules. Re-drop the file into its box after any edit. The box shows the filename and the date it was loaded — check it matches.

What this review will not do

It never touches the question, the bands, the guidance or the wording — that's the quality review's job, and the split is deliberate. Rating an entry and re-authoring it are different acts, and mixing them lets an AI quietly move a threshold to suit the rating it wants to give. If a reply comes back with a changed band, something has gone wrong: reject it.

You can do this without the tool at all

Nothing stops you copying an entry and the contents of entry-quality-rubric.md straight into an AI chat and asking it to check the framing. The buttons just make it repeatable: the right rubric every time, the id preserved, and the reply flowing back into the inbox instead of being retyped by hand.

Chapter 8

Triage — when an entry surfaces

Every other chapter is about what an entry says. This one is about whether a designer ever sees it.

On the live site a designer can run guided triage: they tick what's in the job, answer a few follow-up questions, and the framework hands back the entries worth reviewing. Those suggestions don't come from search or from a category — they come from a trigger you set on each entry, in the Triage map tab. An entry with no trigger is invisible to every designer who arrives that way, however good it is.

The asymmetry that drives everything here

Triage produces a candidate list the reader curates, not a verdict — they tick the ones to keep. So a suggestion they don't need costs them one line to skim. An entry that fails to surface costs them the entry, and they never find out it happened. Bias every call towards surfacing. A slightly loose trigger is cheap; a slightly tight one is silent.

The vocabulary: three kinds of fact

Triggers are built from facts about the job — nothing else. There are three kinds, and there is no fourth:

  • Presence gate — an element is on the job. Ticked at step 1, straight off the architect's drawings. slab-suspended, slab-on-ground, footing, wall, column, pile, beam-suspended.
  • Condition gate — a cross-cutting state, also ticked at step 1. exposed-concrete, water-retaining, wet-or-drained-slab, retained-or-staged.
  • Question — yes / not sure / no, asked at step 2 and only when one of its when gates is ticked. Each is tagged with where the answer comes from, so the reader isn't hunting for it: drawings, geotech, site, or intent (a decision they're still making). reactive-ground, large-bars, considering-screw-piles…

“Not sure” counts as yes. The entry surfaces, flagged “you weren't sure.” That is the asymmetry again: a prompt to check is exactly what a not-sure answer deserves.

How a trigger reads

A trigger is a list of AND groups; inside a group, the facts are OR'd. The builder shows it the same way — chips in a group, the word AND between groups:

"triage": [ ["wall","column","beam-suspended"], ["exposed-concrete"] ]

   =  (wall OR column OR suspended beam)  AND  exposed concrete

Two more fields sit alongside it:

  • Standing entry — a tick-box that removes the trigger entirely. The entry then surfaces on every run, in its own Every job group above the job-specific results. The bar is scope, not importance: the honest scope sentence has to be “on every reinforced-concrete job.” Formwork and falsework complexity qualifies — every RC job has formwork to build and strip. An entry that merely feels important does not.
  • Hide if (triageExclude) — facts that suppress the entry even when its trigger matches. It exists for one job only: routing between two entries that genuinely compete, so the reader gets the right one rather than both.

The status line at the top is the thing to read first — it either says all triggers are valid, or it counts the problems and offers to filter down to them. Search, filter by dimension, or filter by fact (“show me everything that surfaces from reactive ground”).

Fix the vocabulary rather than approximating with the wrong fact.

Switch to the Questionnaire view to edit the gates and questions the designer actually sees: the label and hint on a gate, the prompt on a question, which gates reveal it (when), and where its answer comes from. The one rule

Everything above collapses into a single test, and the rubric is built on it:

Name the jobs the entry bites on.
Then find the smallest set of facts that reproduces exactly that set of jobs.

Then check it three ways. Does it fire on jobs the entry doesn't help? Does it miss jobs it does help — the silent failure, the one worth hunting? And if no fact set reproduces your list, go and fix the vocabulary.

One warning: the entry's own “Applies where” sentence is the obvious place to get that list of jobs, and it is not trustworthy on its own — it was written by the same person who wrote the trigger, so it agrees with the trigger by construction. Read the bands instead, find where the guidance actually changes what a reader does, and name a job the entry would help on that the sentence leaves out.

The trap that hides real defects: the hidden when

A trigger built on a question silently inherits that question's when gates as an extra AND group. If large-bars is only asked when suspended slabs is ticked, then every entry triggered by large-bars is dark on every ground-slab job — including the house with a waffle raft and N20s in the ribs, which is exactly the job those entries were written for. The trigger reads perfectly. The fact exists. The validator is happy. The entry never appears. Always ask: on the jobs this entry names, is its question even asked? The dry run is how you catch it.

Two rules the validator enforces for you

Every “Hide if” owes a receiving entry. If you suppress an entry on a fact, name the entry that picks the reader up instead — and check that its trigger really does fire there. An exclude with nothing behind it is a recall hole that looks like tidiness. And no entry may be both standing and triggered. Standing means universal in scope; a trigger means conditional. Pick one.

Reader appetite is not a job fact

There is deliberately no opt-in in this system — no “tick here to also see the general safety advice.” “I've read this before and considered it” is a fact about the reader, not about the job, and it must never enter a trigger. Default-off toggles fail in the worst direction: the reader who has never considered the advice is precisely the one who never switches it on. Advice is either universal in scope (make it standing) or conditional (give it a gate or a question). Muting is a display choice, and it lives on the site — the standing entries are grouped and collapsible there — never in the data.

The full reasoning behind all of this — worked examples, the precision-vs-recall argument, and how to record the call you made — is in tools/triage-rating-rubric.md. It's also the rubric an AI evidence & triage review (chapter 5) applies when it proposes triggers for you: the suggestions come back to a review screen where you accept them field by field, so the judgement stays yours.

Decision path (by hand)

The one thing the tool doesn't manage: the branching decision path that links the reactive-ground entries, edited directly in data.json.

Almost everything in the framework is managed inside the tools — including the “related entries” links between entries. There is exactly one thing that isn't, and this page is the whole of it. If you never touch the reactive-ground entries, you'll never need this.

What it is

A handful of entries about building on reactive ground form a small branching decision. One entry is the start — it asks the top-level question and points to two follow-on entries, one for each branch. It looks like this on the live site:

How it's stored

It's a small decisionPath object on the data of each entry. The start entry declares itself and names its two branches by their ids:

"decisionPath": {
  "role": "start",
  "groundBearing": "raft-slab-vs-waffle-pod",
  "pileSupported": "pile-supported-beam-and-slab-vs-flat-slab"
}

Each of the two branch entries points back up to the start:

"decisionPath": {
  "start": "found-below-reactive-ground-vs-stiffened-slab-on-top"
}

How to edit it

  1. Open data.json in a text editor.
  2. Find the entry by its id (the same stable id the Data Manager shows).
  3. Add or adjust the decisionPath object on that entry's data, using the shapes above. The ids in it must match real entry ids exactly.
  4. Save, then open the live site (or the Data Manager preview) to confirm the “Decision path” box appears and its links work.

Related-entry “see also” links are not edited here — those are managed in the Data Manager with the Links button. The decisionPath wiring on the reactive-ground entries is the single exception, and this page is all of it.

Which format, and when

WebP is the default for a reason — for a photograph it gives the smallest file at a given quality, and every current browser reads it. But it is a lossy format: it throws detail away to save space, and the first thing to suffer is fine, high-contrast detail — exactly what a bar callout, a dimension string or a scanned note is made of. If you send a drawing through WebP at the default settings, the text goes soft and mushy, and no amount of zooming on the live site brings it back.

So pick by what's in the image, not by habit:

  • WebP — photographs. Site photos, formwork, pours, general condition shots. The default. Quality 0.88 is plenty.
  • PNG — anything with lettering or line work. Screenshots, CAD or Revit extracts, marked-up drawings, diagrams, sketches, tables, anything with small text, thin lines or sharp edges — and anything needing transparency. PNG is lossless: what goes in comes out, text stays crisp. Files are bigger, and that is the correct trade for a drawing nobody can read otherwise. The quality slider does nothing for PNG — it's lossless either way, so don't go hunting for it.
  • JPEG — only when something old needs to open it. It's lossy like WebP but bigger; there's rarely a reason to choose it for the framework.
  • Keep each original format — a mixed batch where the originals were already sensible.

The resize is what usually ruins the text — not the format

If your image has small writing in it, the most damaging setting isn't the codec, it's Resize — max long side. The default 1600 px is right for photographs and will happily shrink a 4000 px drawing down until the callouts are three pixels tall and permanently unreadable. For anything you expect a reader to read, choose 2000 px, or Keep original size. Do this and use PNG — getting one right and the other wrong still gives you a bad image.

A rule of thumb

Can someone read words in it? → PNG, and 2000 px or original. Quality slider irrelevant.
Is it just a photo? → WebP, 1600 px, quality 0.88. Leave everything alone.
A photo where fine detail really matters (a hairline crack, a tight bar clash)? → stay on WebP but push quality to 0.95–1.00 and go up to 2000 px. Note that even at 1.00 WebP is still lossy — if it has to be pixel-perfect, use PNG.

Always check the result before you download: the tool shows you the converted file, so open it and look at the smallest text in it. If you can't read it in the preview, a reader on the live site can't either.

Location data is removed for you

Re-encoding strips the embedded EXIF metadata — including GPS coordinates and camera details — so a site photo can't quietly reveal where it was taken. That happens automatically; you don't have to do anything.

What it checks — and what to do about it

Errors — fix these before you commit

  • Missing images — an entry points at a file that isn't in the folder. It will show as broken on the live site. Add the file (via the Image Formatter) or fix the reference.
  • Entry ids — two entries share an id, or an id is malformed. Ids are how everything else finds an entry, so a clash breaks links and comments.
  • Cross-references — a "related entry" link points at an id that no longer exists (usually something renamed or removed). The link is dead on the site.

Warnings — read these carefully

  • Case mismatches — the reference and the real filename differ only in capitalisation (Detail-A.webp vs detail-a.webp). This is the nastiest bug in the whole system, because Windows and macOS don't care about case, so it works perfectly on your machine — and the server the live site runs on is case-sensitive, so it 404s for everyone else. You cannot catch this by testing locally. Fix every one.
  • Live entries — completeness — a published entry is missing something it should have (question, parameter, bands, applicability). It's live and half-built.
  • Non-web formats — an entry references a format browsers may not display (anything outside PNG, JPG, JPEG, WebP, GIF, SVG). Run it through the Image Formatter.

Notes — housekeeping

  • Internal notes in published file — an entry carries internal working fields (anything starting with _) that would ship in the public file. Worth clearing: the published data.json is world-readable, and this is where a stray private note would leak.
  • Complete but unpublished — a draft that looks finished but is still hidden. Almost always an entry you meant to publish and forgot. Go back to the Data Manager and publish it.
  • Oversized images — a file over 600 KB. Not broken, just slow. Re-run it through the Image Formatter — though if it's a large drawing you deliberately kept at full size so the text stays readable, leave it alone. That's a judgement call, which is why it's only a note.
  • Orphan files — an image in the folder that nothing references. Harmless; delete it when you're tidying.

Last stop before commit

Nothing here changes your files — the Doctor just looks. Run it after you've exported data.json and added your images, as the final check before pushing to GitHub. Zero errors is the bar; the case-mismatch warning is the one that will embarrass you if you skip it.
