Maintainer's manual

Keeping the framework alive

How to review submissions, publish entries and manage images for the Constructability Framework — written for someone who has never opened these tools.

Every task is a real worked example, start to finish, with a picture of what should be on your screen at each step and exactly what to click next.

If you can follow instructions, you can run this.

The big picture

What you're actually maintaining

The framework is a website (index.html) that reads its entire contents from one file — data.json — plus a folder of images. That's it. You never edit the website's code. Your whole job is to keep data.json and the images/ folder correct and up to date.

Three small tools do that work. Each is a single web page you open on your own computer — they run entirely in your browser, upload nothing, and only read or write the files you hand them. Here's how everything fits together:

The routine

The loop you'll repeat

Whenever submissions have built up, you run the same six steps. The rest of this manual is just each of these explained slowly, with a full example. This is the whole job on one screen.

  1. Download the responses In Google Forms, export each form's responses as a CSV — up to three files (new entries, edits, comments).
  2. Work the inbox in the Data Manager Load data.json and the CSVs, then go through each submission: de-identify it, tidy it, and accept or reject it.
  3. Publish the drafts Accepted entries wait as hidden drafts. Review them together, then publish and download the updated data.json.
  4. Prepare the images Run any new photos through the Image Formatter to rename, shrink and strip location data, then drop them in images/.
  5. Run the Doctor Point the Framework Doctor at your updated data.json and images/ folder to catch anything missing before it goes live.
  6. Commit & push Push data.json, the new images and CHANGELOG.md to GitHub. The live site updates itself.

The one thing the tools don't do

Almost everything happens inside the tools — including the “related entries” links between entries, which are fully managed in the Data Manager. There is exactly one exception: the branching decision path that links the reactive-ground entries together. That small piece is written by hand in data.json.

The Data Manager guide has a dedicated section showing exactly what that looks like and how to edit it — so it isn't knowledge that lives only in someone's head.
